# CRM 沟通群设计

## 目标

将客户沟通群从动态数据模板中拆出，作为以案件流程实例为主归属的独立业务对象。首期只内置“企业微信”群类型，但类型保持可配置；同一案件允许保留多个历史群，同时最多只有一个使用中的群。

## 领域边界

沟通群采用“类型配置 + 实际群记录 + 关联人员”三层结构：

- 沟通群类型只描述平台类型，字段为名称、编码、说明、状态和排序。
- 沟通群记录关联流程实例、客户和资产，保存群名称、外部群 ID、状态、建群日期、解散日期、解散原因、智能总结和备注。
- 群关联人员只关联 CRM 人员。工作台手工添加的人员使用“参与人”角色；历史导入可以保留 NPL、PM、ALA 等来源角色。

沟通群不是产品，不启动服务流程，也不复用客户产品表。它只借鉴产品的“可配置类型 + 客户实际记录”结构。

## 业务规则

1. 每条沟通群必须关联一个客户资产流程实例，并同步保存该实例的客户和资产归属。
2. 同一流程实例可以有多个已解散群，但最多一个使用中群。
3. 建群状态中的“待建群”仍属于流程任务；创建沟通群后记录建群事实，不把已解散状态回写为任务未完成。
4. 解散操作只允许从“使用中”转为“已解散”，不支持直接恢复；重新建群应新建记录。
5. 工作台只有当前流程负责人、当前待办办理人、当前部门负责人或具有派单管理权限的人员可以维护沟通群；其他可见人员只读。
6. 新增和编辑时只允许选择启用的群类型和启用人员；已停用的历史关联人员仍可展示。
7. 创建、修改和解散必须写入案件操作记录，并与群记录及人员关系处于同一事务。

## 页面设计

### 工作台

客户/资产详情增加“沟通群”页签：

- 顶部展示“新增沟通群”按钮；无具体流程实例或无维护权限时隐藏。
- 使用中群优先展示，已解散群按建群日期倒序展示。
- 群记录展示平台类型、群名称、建群日期、状态、关联人员、总结和备注。
- 编辑弹窗包含基础信息和关联人员；关联人员复用日程的“添加人员”交互，先选部门，再多选人员。
- 使用中群提供“解散”操作，弹窗填写解散日期和解散原因。

日程和沟通群共同使用通用人员选择组件。日程继续锁定创建人/负责人，沟通群不强制锁定任何人员。

### 后台

客户管理增加“沟通群”全局只读列表，支持按群名称、客户、类型和状态查询。群资料只能从工作台案件详情修改，确保操作记录和关系同步不被绕过。右上角“群类型”进入类型配置弹窗；首期迁移内置“企业微信”。

## API 与数据流

```text
客户/资产详情
  -> CustomerDetail 返回 communication_groups 和维护权限
  -> 打开编辑弹窗时读取共享 people_options
  -> save_communication_group / dissolve_communication_group
  -> 领域校验 + 事务保存群及关联人员 + 操作记录
  -> 触发工作台统一刷新
```

日程的 `schedule_options` 与沟通群的 `people_options` 共用服务端人员、部门选项构造函数，避免两套人员筛选规则。

## 飞书历史导入

“企业微信群”来源表改为写入沟通群对象：

- `案件ID`定位流程实例，并取得客户和资产归属。
- `群名称`、`创建日期`、`智能总结`、`解散日期`、`解散原因`、`备注`写入群记录。
- `NPL负责人(人员)`、`DPM负责人`、`ALA负责人`写入群关联人员并保留来源角色。
- `目前阶段`和`是否签约`只做流程对账。
- 飞书表 ID 与 record_id 组成来源键；完全重复业务记录合并，全空记录跳过并保留导入审计。

## 验证边界

按项目要求不运行 build 或自动化测试。实现后仅执行 `gofmt -d`、JSON 解析、Dever 静态审计、`git diff --check`和源码引用检查，由用户手工验证页面与业务链路。
