# 客户标签自动判级与来源继承设计

## 目标

在签约流程的客户资料表单中完成两项调整：

1. 来源、渠道沿用来源线索的数据，并在客户资料表单中自动带入。
2. 工作人员先选择客户标签，系统根据标签所属分组自动确定客户等级。

本轮只实现四级客户标签及自动判级，不引入通用规则引擎，不把标签放入数据模板，也不增加独立的标签规则配置页面。

## 配置边界

现有 `crm_customer_level` 继续作为客户等级配置，当前四个等级为：

- 高意向。
- 中意向。
- 无意向。
- 联系不上。

标签直接归属于客户等级。后台在现有“客户等级”编辑页内维护该等级的标签，管理员可配置标签名称、状态和排序。

客户尚未选择标签时使用 `level_id=0` 表示未判级，不额外创建“普通”业务等级。迁移不会改写历史客户等级；旧“普通”等级只从活动配置中停用，客户选择新标签后再进入四级之一。

数据模板仍只负责客户、资产等扩展资料采集。客户标签会参与客户等级判定并影响客户主表，因此不能作为普通数据模板字段或常用选项集处理。

## 数据模型

新增客户标签模型 `crm_customer_tag`：

- `id`：标签 ID。
- `level_id`：所属客户等级。
- `name`：标签名称。
- `status`：启用状态。
- `sort`：同等级内排序。
- `created_at`、`updated_at`：审计时间。

新增客户标签关联模型 `crm_customer_tag_relation`：

- `customer_id`：客户 ID。
- `tag_id`：标签 ID。
- `created_at`：关联时间。

同一客户与同一标签只能存在一条关联。标签名称可以调整，客户关系始终使用稳定的标签 ID。

现有 `crm_customer.tags` 不再作为标签判定依据。为兼容当前列表、详情和导出展示，保存标签关系时同步写入标签名称快照；真实关系以关联表为准。现有自由文本标签不自动猜测映射，客户下次重新选择标签后再切换到结构化关系。

## 标签选择与等级判定

签约表单的“标签”改为分组单选控件，按四个客户等级展示对应标签。

选择规则固定为：

1. 一个客户只能选择一个标签。
2. 选择新标签时直接替换原标签，无论两个标签是否属于同一等级。
3. 标签按钮使用单选语义，并以高对比实底样式显示当前选择。
4. 标签变化后，表单中的客户等级立即显示为当前标签所属等级。
5. 客户等级只读，不允许工作人员单独修改。

后端保存时再次校验只能提交一个启用标签。校验通过后，在同一业务保存过程中同步客户标签关系、标签名称快照和 `customer.level_id`；多标签请求直接拒绝保存，不能依赖前端限制保证数据正确。

签约流程默认表单将标签设为必填。其他任务是否要求填写标签，继续由资料模板的必填配置决定；没有提交新标签时，不主动改写客户当前等级。

## 来源与渠道继承

来源、渠道继续使用线索和客户主表已有的 `source_id`、`channel_id`，不新增重复字段。

数据链路为：

1. MKT 在线索确认任务中录入或修正来源、渠道。
2. 线索转客户时，将线索的 `source_id`、`channel_id` 写入新客户。
3. 签约任务打开时，从客户详情加载当前来源、渠道作为表单初始值。
4. 默认签约表单将来源、渠道配置为只读，避免 NPL 修改 MKT 归因；后续是否允许修改仍可在资料模板中调整只读配置。

当前图示客户对应的数据库记录中，线索与客户均为“手动录入 / 抖音”，说明现有转化链路已经完成复制。若业务期望其他值，应修正 MKT 线索录入或同步数据，不能在签约表单中猜测来源和渠道。

## 后台配置

“客户等级”编辑弹窗增加标签列表，复用标准 `form-array` 维护标签：

- 新增标签。
- 修改标签名称。
- 启用或停用标签。
- 拖动调整排序。

标签 CRUD 使用 Model 和现有 page JSON 能力。等级与标签的批量保存、关联校验属于跨表一致性，复用 CRM setting Provider/Service 处理，不新增 CRUD API。

## 工作台实现

任务表单仍使用现有资料模板字段。客户主字段 `tags` 改为客户标签选择类型，服务端给任务字段附加按等级分组的启用标签选项。

工作台增加一个聚焦的客户标签选择组件，负责：

- 展示四个等级及各自标签。
- 全局单选，选择新标签时替换原标签。
- 使用明显的选中样式回显当前标签。
- 回显客户已有的结构化标签。
- 实时显示自动判定的客户等级。

表单提交继续走现有工作任务保存接口；不新增标签专用 work API。

## 异常处理

- 标签不存在、已停用或不属于有效客户等级时拒绝保存。
- 标签输入包含非数字或无效 ID 时拒绝保存，不能按空选择处理。
- 同一次提交包含多个标签时拒绝保存，并提示只能选择一个客户标签。
- 必填任务未选择标签时不能完成，但仍遵循现有“保存进度”能力。
- 标签配置被停用后，历史客户详情仍显示已保存的名称快照；重新编辑时必须选择当前启用标签。
- 来源线索不存在时保留客户当前来源、渠道，不写入默认猜测值。

## 验收标准

1. 后台可在四个客户等级下分别维护标签。
2. 签约表单按等级分组展示启用标签。
3. 标签只能单选，选择新标签会替换原标签，且选中状态清晰可辨。
4. 选择标签后客户等级即时变化且不可手工修改。
5. 保存后客户标签关系、标签展示名称和客户等级一致。
6. 构造多标签请求时后端拒绝保存。
7. 线索转客户后，签约表单自动带入线索来源和渠道。
8. 默认签约表单中的来源、渠道为只读。
9. 不新增数据模板字段、通用规则配置或标签 CRUD API。

按项目要求不运行构建命令或自动化测试，页面和完整业务链路由用户手工验证。
