# CRM 客户工作区体验优化设计

日期：2026-07-12
状态：已确认
目标分支：`feature/customer-workspace-ux`

## 1. 背景

当前 work 前台已经具备客户列表、客户/资产详情、动态任务表单、资料进度保存和流程记录等完整能力，但页面组织方式仍影响日常使用效率：

- 客户列表默认进入“待处理”，没有待办时页面看起来像没有客户。
- 六个搜索框平铺，常用搜索需要先判断字段，再选择输入框。
- 表格操作列按任务数量纵向增长，行高和横向宽度都不稳定。
- 客户详情的总览与资料区域重复，空字段被完全隐藏，无法快速判断还缺什么。
- 长资料表单与普通事项共用同一种纵向表单布局，十一维资料需要大量滚动。
- 分组多选、布尔、附件等字段缺少适合业务录入的控件表现。
- 抽屉内流程记录采用左右交错时间线，宽度利用率较低。

本次改造不调整客户、资产、流程、阶段、任务和数据模板模型，只优化已有数据的查找、录入和展示路径。

## 2. 设计目标

1. 客户列表打开后立即看到可访问客户，不因当前没有待办而显示空页。
2. 常用搜索只需要一个输入框，精确条件仍可使用高级筛选。
3. 表格行高稳定，一行只保留一个明确的主要动作。
4. 普通任务保持轻量，长资料表单支持分组填写和多次保存。
5. 详情抽屉同时回答“客户是谁、现在到哪一步、缺什么、下一步做什么”。
6. 所有表单和详情仍由现有流程、任务、资料模板配置驱动，不新增第二套配置。
7. 保持现有 work API、页面节点名和保存语义，控制改动范围。

## 3. 非目标

- 不新增客户状态字段，不把阶段复制到客户主表。
- 不新增通用 CRM 页面生成器、表单引擎或工作流引擎。
- 不允许绕过任务直接修改受流程控制的业务资料。
- 不加入自动保存，避免网络抖动或多人协作时产生隐式覆盖。
- 不重做后台数据模板、资料模板或流程配置页面。
- 不修改 `package/front`、Dever runtime 或编译产物。

## 4. 方案选择

### 方案 A：只调整视觉样式

保留现有信息结构，仅调整间距、字号和颜色。改动小，但不能解决空列表、任务堆叠、长表单滚动和详情重复问题。

### 方案 B：紧凑列表 + 详情工作区 + 自适应任务表单

复用现有业务数据和任务执行接口，重新组织列表、详情和表单的呈现方式。少量扩展现有只读响应，以支持综合搜索和完整资料展示。

### 方案 C：改成独立客户详情页面

为每个客户建立完整路由、子页面和编辑体系。扩展空间最大，但会引入新的导航、权限和保存路径，超出当前“简单易用”的目标。

采用方案 B。它能解决主要使用问题，同时避免新增模型和重复业务能力。

## 5. 客户列表

### 5.1 默认状态和顶部筛选

- 直接进入客户列表时默认模式为“全部”。
- 从工作台指标或待办下钻进入时，继续尊重 URL 中的 `mode`、阶段和任务筛选。
- 顶部模式顺序为“全部 / 待处理 / 已结束”，每项显示数量。
- 有调度权限的账号继续显示“我的 / 全部”范围切换。
- 刷新保留为图标按钮，并提供 `刷新` 提示。

### 5.2 搜索

主工具栏只展示：

- 综合搜索：姓名、手机号、微信号、客户编号、资产编号。
- 当前阶段：可清除的选项菜单。
- 更多筛选：展开精确字段，包括客户编号、姓名、手机号、微信号和资产编号。
- 搜索与重置。

前端直接复用现有 `/crm/work/customers` 的 `keyword` 参数；服务已经按上述五类信息执行 OR 匹配，无需新增搜索接口或业务逻辑。现有精确参数继续保留，避免影响工作台下钻和旧 URL。

客户列表响应补充当前可用阶段选项，供前端筛选菜单使用；不新增独立 options API。

### 5.3 表格结构

桌面端固定为五列：

| 列 | 内容 |
| --- | --- |
| 客户 | 姓名、客户编号、手机、微信 |
| 房产/资产 | 资产名称、资产编号、地址或“未录入资产” |
| 流程阶段 | 流程名、阶段名、负责人、停留天数 |
| 当前待办 | 首要任务、任务数量、无待办状态 |
| 操作 | 全部可处理任务按钮、改派和详情 |

行为规则：

- 点击客户或整行空白区域打开详情。
- 有可处理任务时，按配置顺序直接展示全部可处理任务按钮。
- 没有任务时，主要按钮为“查看”，打开详情。
- 多个任务在操作列内自动换行，不收进“更多”菜单。
- 规则任务显示为只读核验状态，不伪装成可点击按钮。
- 表格行使用稳定的最小高度，文本超长时最多显示两行。

移动端继续使用列表卡片，内容顺序与桌面一致，全部任务按钮在操作区内自动换行。

## 6. 自适应任务表单

### 6.1 两种呈现模式

任务打开前根据现有动态节点判断表单规模：

- 紧凑模式：普通事项、审核或不超过 6 个简单字段，使用当前居中弹窗。
- 工作区模式：超过 6 个字段、包含多个资料分组或包含 P01-P12 分组，使用接近全高的宽弹窗。

该判断仅影响展示，不改变任务类型、提交参数或权限。

### 6.2 工作区结构

长表单由三部分组成：

1. 固定顶部：任务名、客户、资产、当前阶段、必填完成数量。
2. 中间内容：左侧分组目录，右侧当前分组字段。
3. 固定底部：AI 填写、保存进度、确认完成。

P01-P12 直接复用资料模板中已有的 group 字段作为目录。普通非 group 字段继续按客户信息、资产信息、补充信息分区，不新增分组配置。

### 6.3 字段布局

- 输入框、日期、数字、单选等短字段在桌面端两列展示。
- 文本域、附件、复杂多选占整行。
- 布尔字段使用开关或明确的“是 / 否”选择，不渲染成文本输入框。
- 多选字段使用可扫描的复选选项或多选菜单，不使用原生多选列表框。
- 附件区域保留上传、预览、下载和数量限制能力。
- 必填标记、错误文本和禁用状态使用统一字段容器。
- 移动端全部切换为单列，分组目录变成横向可滚动标签。

### 6.4 保存和校验

- “保存进度”允许缺少必填项，继续复用现有 `submit_mode=progress`。
- “确认完成”执行完整校验，继续复用现有 complete 提交。
- 校验失败时，在表单顶部显示缺失字段摘要；点击条目切换到对应分组并聚焦字段。
- 保存期间锁定重复提交，失败时保留当前输入。
- 不做自动保存。

## 7. 客户详情抽屉

### 7.1 顶部上下文

保留右侧详情抽屉，不新增详情路由。抽屉顶部固定展示：

- 客户姓名和状态。
- 客户编号、手机号、当前资产编号。
- 当前流程、当前阶段、负责人和停留天数。
- 有待办时显示“处理当前任务”主要按钮。

### 7.2 总览

总览只保留决策信息：

- 当前阶段和当前待办。
- 客户关键联系方式。
- 当前资产关键字段。
- 资料总体完整度和缺失最多的模板。
- 最近 5 条操作。

不在总览重复完整资料明细。

### 7.3 资料

资料页按目标对象和模板分组：

- 客户基础信息。
- 客户资料模板。
- 当前资产基础信息。
- 资产资料模板，包括十一维资料。
- 已生成的业务对象资料，如租赁、产品或合同记录。

每个模板显示完成比例、已填数量和总数量。字段统一显示为标签和值，空值明确显示“未填写”，附件显示文件数量并允许预览。

为此，现有 `/crm/work/customer_detail` 响应增加仅供详情使用的 `detail_sections` 投影。该投影复用现有数据模板字段、字段显示转换和资料记录，不新增 Model、Service 或 API 路由。

### 7.4 流程

- 当前阶段、负责人和待办任务置顶。
- 操作记录改成单列时间线，按时间倒序。
- 保留“全部记录 / 我的记录”切换。
- 点击记录继续打开现有任务记录详情。
- 阶段完成、负责人调整和终止等现有流程动作保持原权限和行为。

## 8. 前端结构与复用

当前 `work-auth.tsx` 同时承担登录、统计、客户列表、详情、任务表单和资料控件职责。本次只拆分直接涉及的三块：

- `work-customer-list.tsx`：筛选、表格、移动卡片、分页。
- `work-customer-detail.tsx`：详情抽屉、总览、资料、流程记录。
- `work-task-form.tsx`：动态表单布局、字段容器、进度和校验摘要。

共享类型、API、格式化函数和 store 访问继续放在现有 `work-core.ts` 或小型共享模块中。插件节点名保持不变，由 `plugin.ts` 继续懒加载公开节点，避免 page JSON 改动扩大。

拆分过程中不顺手重构登录、统计、线索或后台页面。

## 9. 数据流

### 客户列表

1. 页面从 URL 初始化模式和筛选。
2. 前端调用现有 `/crm/work/customers`。
3. 服务在现有权限范围内完成 keyword、精确字段、阶段、任务和 scope 过滤。
4. 前端把客户与资产转换成稳定行模型。
5. 点击任务打开现有动态任务表单；点击其他位置打开详情。

### 详情

1. 前端使用列表行中的客户/资产快速打开抽屉。
2. 继续调用 `/crm/work/customer_detail` 刷新完整内容。
3. 服务返回客户、资产、流程、待办、操作和 `detail_sections`。
4. 任务完成后通过现有 `crm-work-refresh` 事件刷新详情和列表。

### 表单

1. 继续使用任务中的 form 和 fields 生成动态节点。
2. 表单布局根据字段数量和分组切换紧凑或工作区模式。
3. 保存进度和确认完成继续调用 `/crm/work/execute`。
4. 后端继续负责权限、必填校验、资料保存和流程副作用。

## 10. 错误和空状态

- 列表加载失败保留当前筛选，并显示可重试提示。
- 综合搜索无结果时显示当前筛选摘要和“清除筛选”。
- 详情加载失败保留基础客户信息，并提供重试按钮。
- 资料模板没有字段时显示“暂无配置字段”，不显示空白卡片。
- 表单提交失败不关闭弹窗，不清空已填写内容。
- 用户无任务权限时只显示详情，不显示“处理”按钮。

## 11. 响应式与可访问性

- 桌面端以 1440px 以上为主要密度目标，1280px 仍不出现页面级横向滚动。
- 移动端列表改为卡片、详情抽屉占满屏、表单字段单列。
- 所有图标按钮有可访问名称和悬停说明。
- 标签与输入控件建立明确关联；键盘可以完成筛选、切换分组和提交。
- 固定头尾不能遮挡表单字段或错误提示。

## 12. 验收标准

1. 直接打开客户列表能看到“全部”客户。
2. 一个综合搜索框能命中姓名、手机、微信、客户编号和资产编号。
3. 多任务客户的表格行不因任务数量显著增高。
4. 列表主要动作能直接打开当前优先任务。
5. 普通事项和审核仍以紧凑弹窗完成。
6. 十一维表单可以按 P01-P12 分组切换、保存进度和继续填写。
7. 完成校验能定位到缺失字段。
8. 详情资料页能展示已填值和“未填写”字段。
9. 详情流程页使用单列时间线，现有流程动作可用。
10. 桌面和移动端没有页面级横向溢出，控制台没有新增错误。

## 13. 验证方式

遵照项目要求，不运行 `npm run build` 或任何测试命令。实施后使用已启动的 Dever 环境，通过浏览器逐项验证：

- 综合搜索和精确筛选。
- 全部、待处理、已结束和范围切换。
- 无任务、单任务、多任务和规则任务行。
- 普通事项、审核、短资料表单和十一维长表单。
- 保存进度、校验失败和确认完成。
- 客户详情三个标签及任务记录详情。
- 1920px 桌面和 390px 移动视口。
