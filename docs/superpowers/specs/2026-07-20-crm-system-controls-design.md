# CRM 系统控件收敛设计

> 架构更正：本文原有的任务级财务/统计方案已由
> `2026-07-20-task-capability-simplification-design.md` 取代。当前实现中会议和下次跟进属于任务能力，财务类型和参与统计属于数据字段属性；旧任务关系仅保留审计，不参与运行时。

## 目标

财务登记、客户下次跟进、产品选择、会议预约和任务统计都归任务配置，不再通过“系统用途”建立第二层映射。数据模板仅保存客户、资产和业务资料；任务配置声明办理行为及需要统计的表单字段。

## 任务配置

- 任务与财务类型使用 `TaskFinanceType` 多对多关系。后台任务高级配置直接多选需要登记的财务类型。
- 任务与统计字段使用 `TaskStatField` 多对多关系，只允许选择当前任务表单中的启用数据字段。
- 表单任务可开启“需要填写下次跟进时间”。开启后办理页自动出现固定日期时间控件。
- 会议继续使用现有 `meeting_enabled` 开关；产品继续使用现有产品任务类型，两者不再扩展数据模板映射。
- 所有运行时逻辑只依赖任务 ID、财务类型 ID 和开关，不依赖任务名称、字段名称或固定人员。

## 财务登记

任务办理页根据任务已配置的启用财务类型生成金额控件，字段键使用 `finance:<finance_type_id>`。金额在任务最终提交时直接写入 `FinanceLedger`，不写入 `DataRecord`。

`FinanceLedger` 增加稳定的 `finance_source_key`：历史数据回填为 `field:<data_field_id>`，新系统控件写为 `finance:<finance_type_id>`。幂等键改为“流程实例 + 操作记录 + 来源键 + 来源”，避免新记录的 `data_field_id=0` 相互冲突。

现有有效财务映射迁移为任务财务类型关系，原数据模板财务字段从对应任务表单停用，历史 `DataRecord` 和 `FinanceLedger` 不删除。已失效或语义已经变化的旧字段映射不迁移。

## 客户跟进

任务开启跟进开关后生成 `follow_up:start_at` 固定控件。最终提交直接创建或调整该客户唯一待进行跟进日程；空值会取消原待进行跟进日程。日历中新建、改期、完成和安排下一次也只操作 `ScheduleEvent`，不再反写客户数据模板。

旧日程原有时间、负责人、参与人和操作记录全部保留；只清空已无意义的数据用途/数据字段关联 ID。

## 系统用途退役

- 数据模板页移除“系统用途”入口和相关管理页面。
- 删除财务、客户跟进、展示和统计用途的运行时读取；统计写入改由任务统计字段关系驱动。
- `StatFieldValue`、`DataRecord`、`FinanceLedger` 等历史业务数据保留，不批量删除。
- 旧用途配置表暂留数据库作为迁移审计数据，但应用不再注册模型、页面或服务，也不再读取和写入。

## 配置界面

- 基础配置直接显示任务名称、类型、表单、负责人和期限。
- 高级配置分为“流程控制”和“业务能力”。
- 财务类型和统计字段直接使用标准多选控件，不增加二次配置弹窗。
- 预约会议、下次跟进和负责人参会使用短标签；到访确认由预约会议自动启用，不单独配置。

## 字段统计

- 任务提交资料时，只把本任务已选择且本次实际提交的数据字段同步到 `StatFieldValue`。
- 字段类型自动映射统计类型：数字为数值、金额为金额、日期时间为时间、开关为状态，其余为文本或分类。
- 业务看板读取统计快照，按流程和日期范围展示有效记录、数值汇总、时间范围和值分布。
- 同一线索或客户、资产、流程实例和字段继续保持一份最新快照，不删除已有历史业务数据。

## 迁移原则

迁移只负责把当前有效配置转换成新任务配置，并停用被系统控件替代的表单字段。运行时代码不使用任务名、表单名或字段 key；这些标识只允许在一次性迁移中定位现有配置。

## 验收边界

- 任务高级配置能维护财务类型、统计字段和下次跟进开关。
- 办理任务时自动显示已配置金额项或下次跟进时间，不要求数据模板中存在对应字段。
- 提交后分别写入财务流水和日程，重复处理同一操作记录不重复写。
- 数据模板页不再显示“系统用途”，历史流水、历史资料和既有日程仍可查看。
- 按项目要求不执行 build 或 test，由用户手工完成页面和完整流程验收。
