# CRM 签约主链重整设计

## 目标

先跑通“线索转客户 -> 接单建档 -> 资料收集 -> 自动诊断 -> 产品确认 -> 多部门协作 -> 合同签署”主链。

本轮只处理默认入口流程，不改运营流程，不实现一个签约流程内的多合同、续签和合同版本管理，也不兼容旧资产进度和旧待办数据。

## 三层配置职责

保留现有三层配置，不新增平行配置系统：

1. 数据模板只定义扩展字段及其数据归属。
2. 资料模板组合主表字段和扩展字段，并定义当前任务的必填、只读和排序。
3. 流程配置只定义流程、阶段、任务、负责部门和分配方式。

产品是独立核心模型，产品确认必须写入 `CustomerProduct`，不得再用动态字段保存产品编码。

本轮合同先作为归属流程实例的业务数据保存。一条入口流程实例维护一份合同登记；需要一条流程多份合同时再增加独立合同模型。

## 数据模板

固定保留四个数据归属分类：

- 线索信息：线索主表之外的补充字段。
- 客户信息：客户主表之外的跟进和建档字段。
- 客户资产：资产基础资料、十一维事实和诊断结果。
- 业务数据：专业协作、合同和后续流程产生的数据。

签约主链使用以下模板：

- 线索补充信息。
- 接单与建群。
- 资产基础信息。
- 十一维资料。
- 诊断结果。
- 专业协作意见。
- 合同信息。

停用或删除以下重复定义：

- 客户来源与基础建档：与线索、客户主表重复。
- 签约方向候选、签约方向置信度、PM确认签约方向和最终签约方向。
- NPL候选产品编码、NPL主推产品编码和PM确认产品编码。

产品候选和确认使用客户产品表，不再存放在客户资产扩展字段中。

## 资料模板

### 接单建档

一个表单组合以下内容：

- 客户姓名、手机号、微信号、身份证号和备注。
- 首呼结果、建群状态、客户意向、授权状态、首呼摘要和下次跟进时间。
- 资产名称、资产状态、资产城市、资产地址、面积户型、权属说明和基础资料附件。

线索转化时创建的临时资产在此表单中补全，不再保留一个未被流程引用的资产建档表单。

### 十一维资料收集

十一维仍按 P01-P12 分组展示。每组八个字段中，仅“探针选项”是完成任务的必填字段；原话、摘要、证据、附件、缺失项、风险等级和复核备注为可选字段。

业务人员可以反复“保存进度”。只有十二个探针选项全部填写后才能“提交并完成”。

资料模板直接选择分组子字段并分别设置必填。工作台根据字段的父分组自动恢复 P01-P12 标签布局，避免把整个分组设为必填后强制填写全部 96 个字段。

### 专业协作与合同

律师、ALA 和财务继续使用流程任务处理。合同信息模板移到“业务数据”，使合同登记归属当前签约流程实例，而不是覆盖资产级数据。

## 默认入口流程

线索不是流程阶段。只有待处理线索转为客户后，系统才创建客户、临时资产和默认入口流程实例。

默认流程配置为：

1. 接单建档：NPL 负责，必做表单“接单建档”。
2. 资料收集：PM 负责，必做表单“十一维资料收集”，支持保存进度。
3. 诊断核验：PM 负责，先自动执行 T 节点规则，再由 PM 确认诊断结果。
4. 产品确认：PM 负责，使用 `product` 任务生成或确认客户产品。
5. 签约协作：PM 主责，律师、ALA、财务任务并行分配。
6. 合同签署：合同法务部负责，填写合同信息；阶段完成即完成入口流程。

不再保留单独的“签约确认”阶段。正式签署状态是合同表单的必填字段。

任一阶段可以终止流程并记录流失原因。阶段和协作任务继续支持自动分配、手动分配以及流程调度员改派。

## 自动规则输出

规则脚本在现有 `{ passed, value, reason }` 基础上支持两个可选输出：

```javascript
return {
  passed: true,
  value: "T6",
  reason: "已形成可执行依据",
  fields: {
    candidate_t_node: "T6",
    candidate_t_confidence_level: "high"
  },
  product_codes: ["S01", "S03"]
};
```

- `fields` 按全局唯一字段编码写入对应客户、资产或业务数据记录。
- `product_codes` 同步为当前入口流程的候选客户产品。
- 产品确认任务将候选产品预选给 PM，最终以 PM 提交结果为准。

当前资料只给出了 T 节点与产品方向口径，没有完整且唯一的 T-to-S 映射。因此本轮实现候选产品输出能力，但不在代码中猜测或写死产品映射；未配置 `product_codes` 时 PM 仍可手动选择产品。

## 配置落库

`migrations/postgres` 当前没有自动执行入口，现有数据库也未完整应用线索和流程迁移。本轮复用现有线索迁移 `011_lead_data_templates.sql`，再增加一份幂等的签约配置迁移，并在当前环境按顺序显式执行。

迁移负责：

- 先由迁移 011 补齐线索分类和线索补充模板。
- 清理无所属流程的旧阶段和任务。
- 重组数据模板、资料模板和默认入口流程。
- 修复产品分类及资产运营产品的签约后流程。
- 清理旧流程实例、旧待办和客户产品运行数据，不删除客户、资产和线索主数据。

## 验收链路

使用一条新线索验证：

1. 录入线索并转客户。
2. 自动生成客户、临时资产和接单建档待办。
3. 完成接单建档并流转给 PM。
4. 多次保存十一维资料，缺少任一探针选项时不能完成。
5. 完整提交后自动得到 T 节点。
6. PM 确认产品并生成客户产品。
7. 律师、ALA、财务完成协作任务。
8. 合同部门登记合同并完成入口流程。
9. 有签约后流程的产品启动对应流程，无签约后流程的产品直接完成。

按项目要求不运行自动化测试或构建命令，由用户手工验证页面和业务链路。
