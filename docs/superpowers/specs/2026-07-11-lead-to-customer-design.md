# 线索转客户精简设计

## 目标

先完成一个可用的最小闭环：

```text
MKT 手工录入线索
→ 自动检查重复
→ 判无效 / 确认重复 / 转为客户
→ MKT 使用现有任务派单至 NPL
```

线索必须独立于客户保存，确保无效线索、重复线索和已转化线索都能参与来源与转化统计。

## 首版范围

首版只实现：

- MKT 工作台手工录入线索。
- 线索列表、搜索和状态筛选。
- 按手机号、微信号检查已有客户和已有线索。
- 标记无效、确认重复、恢复待处理、转为客户。
- 无效原因可配置，并允许填写补充说明。
- 转化后继续使用现有“派单至 NPL”任务。

首版不实现：

- Excel 导入和外部渠道 API。
- 自动合并客户资料。
- 线索自动派单。
- 独立操作日志表和统计大屏。
- 历史客户反向生成线索。

## 数据模型

### Lead 线索

固定字段只保存查询、去重和状态流转必需的数据：

- 线索编号、姓名、手机号、微信号。
- 来源、渠道、外部线索 ID、城市、初始诉求。
- 状态：待处理、无效、重复、已转化。
- 疑似重复线索 ID、疑似重复客户 ID、重复依据。
- 无效原因 ID、无效补充说明。
- 转化客户 ID、MKT 负责人、创建人、转化人。
- 录入原始值 `record_json`、创建时间、更新时间、转化时间。

手机号和微信号不设置唯一约束，因为重复线索也必须入库。线索编号设置唯一约束。

### LeadInvalidReason 无效原因

普通配置表，只包含名称、状态和排序。默认提供空号/错号、测试数据、非目标客户、区域不符等选项，管理员可自行增删停用。

## 录入方式

复用现在“创建客户与来源登记”的可配置资料模板，把工作台按钮改名为“录入线索”。

提交时不再直接创建客户，而是：

1. 校验姓名以及手机号/微信号至少一个。
2. 保存固定检索字段和完整 `record_json`。
3. 检查相同手机号、微信号或来源外部 ID。
4. 保存疑似重复对象和依据，但仍保留本次线索。

这样以后调整客户扩展字段时，不需要同时修改一套写死的线索表单。

## 状态操作

- 待处理：可以转客户、判无效或确认重复。
- 无效：必须选择无效原因，可以恢复待处理。
- 重复：关联已有客户或已有线索，不创建新客户，可以恢复待处理。
- 已转化：只读，关联唯一的转化客户。

发现疑似重复时禁止直接转客户。MKT 必须确认重复，或者修正联系方式后重新检查。

## 转客户

转化使用数据库事务，并直接复用现有客户创建逻辑：

1. 再次检查手机号和微信号，防止并发产生重复客户。
2. 使用线索 `record_json` 调用现有客户资料模板解析与校验。
3. 创建客户、客户成员、客户阶段和原有操作记录。
4. 在线索上写入客户 ID、转化人、转化时间，并更新为已转化。

转化只生成客户，不自动派单。客户保留在 MKT 阶段，继续执行现有“派单至 NPL”任务。

## 权限和页面

- 只有 MKT 部门可以录入和处理线索。
- 其他部门继续使用现有客户工作台，不显示线索池。
- MKT 工作台在客户列表前增加一个简洁的线索池区域。
- 后台增加无效原因维护页和只读线索列表。

## 错误处理

- 联系方式为空、无效原因为空、线索状态不允许操作时，服务端拒绝提交。
- 转化前再次命中客户时，更新重复提示并拒绝创建客户。
- 已转化线索重复点击时返回原客户，不重复创建。
- 所有状态修改和客户创建放在事务内，避免只转化一半。

## 验收标准

1. MKT 能录入一条正常线索，并在线索池中看到。
2. 相同手机号再次录入时保留新线索并提示疑似重复。
3. 重复线索可关联已有客户，且不能创建第二个客户。
4. 线索可按可配置原因标记无效并恢复。
5. 正常线索可转为客户，客户资料和扩展资料完整保留。
6. 转化后仍能执行原有“派单至 NPL”任务。
7. 非 MKT 人员看不到线索处理入口。

按项目要求，开发过程中不运行 `npm run build` 或测试命令，完成后使用浏览器按上述流程手工验证。
