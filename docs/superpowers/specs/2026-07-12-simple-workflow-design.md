# CRM 简化流程设计

## 1. 目标

只重构 CRM 的流程运行部分，最终只保留五个核心概念：

1. 流程
2. 阶段
3. 任务
4. 当前负责人
5. 流转记录

客户、资产、资料模板、自动规则、产品、合同和运营记录继续作为独立业务数据存在。流程只负责编排这些业务动作，不把它们改造成动态表，也不在客户表重复保存一个“客户阶段”。每项资产拥有自己的当前流程和阶段。

本次不恢复旧版复杂的条件分支、阶段转换规则、多套自动分配策略和协作完成策略，也不使用 ReactFlow。当前业务是有序阶段，列表式配置更直观。

## 2. 方案选择

### 方案 A：恢复旧工作流引擎

可以直接获得分配、协作和转换能力，但旧代码包含大量触发器、条件、完成模式和配置 JSON，正是当前配置和工作台复杂的主要原因，不采用。

### 方案 B：有序阶段 + 自动均衡分配（采用）

流程内阶段按排序依次执行。当前负责人点击“完成阶段”后立即流转，系统默认把业务分给下一部门当前在办最少的人员；需要时可以改为手动指定。它覆盖当前业务，又不会重新引入规则引擎。

### 方案 C：ReactFlow 图工作流

适合并行网关、条件分支和回路。当前只有签约、运营等线性流程，图编辑会增加节点、连线、发布版本和运行解释成本，暂不采用。

## 3. 配置模型

### 流程

- 可新增、编辑、停用和删除，名称完全自定义，例如“签约流程”可改为“签署流程”。
- 配置名称、排序、状态、“默认入口”和可选的“完成后流程”。
- 同一租户只能有一个启用的默认入口。线索转化等自动入口从这里启动；手动启动时仍可选择其他启用流程。
- “完成后流程”用于连接线性流程，例如签署流程完成后进入运营流程；为空表示整个业务流程完成。
- 禁止流程指向自己或形成循环；被其他流程引用时不能直接删除。

### 阶段

- 阶段属于一个流程并按排序执行。
- 配置名称、负责部门、排序和状态。
- 负责部门只是下一位负责人的可选范围，不代表部门内所有人共同负责。
- 部门使用现有部门 ID，不在代码中写死 MKT、NPL、PM、律师或 ALA。

### 任务

- 任务属于一个阶段。
- 保留四种动作类型：普通事项、填写资料、审核、自动核验。
- 配置名称、类型、必做、负责人方式、期限、排序和状态。
- “跟随阶段负责人”的任务属于主流程，直接分配给当前阶段的具体负责人。
- “指定部门”或“指定人员”的任务属于协作，不改变主流程负责人。
- 填写资料只关联现有资料模板；自动核验只关联现有脚本规则，不复制表单和规则能力。

## 4. 运行规则

### 进入首阶段

线索转客户时，调用流程启动服务并传入客户、最小资产和默认入口流程。系统默认从首阶段负责部门中自动分配具体接收人，转化人也可以在提交前改为手动指定。该人员成为首阶段负责人，同时创建首阶段待办。线索池和线索转化页面本身不在本次重构范围内，只调整其调用流程服务所需的接口契约。

从其他入口启动流程时使用相同规则，不创建“只有部门、没有负责人”的活跃流程。

### 执行任务

- 主任务只允许当前阶段负责人处理。
- 指定人员的协作任务只允许该人员处理。
- 指定部门的协作任务允许该部门的有效人员处理。
- 自动核验在所需资料满足后执行，结果写入任务结果；它可以完成任务，但不能自动切换阶段。

### 完成阶段并流转

只有当前阶段负责人可以发起流转：

1. 校验当前阶段所有必做主任务和必做协作任务均已完成。
2. 当前流程还有下一阶段时，确定下一阶段的负责部门。
3. 当前流程已到最后阶段且配置了“完成后流程”时，确定下一流程首阶段的负责部门。
4. 默认自动选择该部门中启用且当前负责进行中资产数量最少的人员；数量相同时，优先选择最久未被分配的人员。
5. 当前负责人也可以在完成阶段前改为手动指定，候选人仍然只能来自下一阶段负责部门，并显示每个人的当前在办数量。
6. 如果下一部门没有启用人员，则拒绝流转并给出明确提示，不进入部门公池，也不产生无人负责的待办。
7. 在一个事务中取消当前阶段未完成的选做任务、记录流转日志、更新流程/阶段/负责人、创建下一阶段任务并授予接收人查看权限。
8. 当前流程没有“完成后流程”时，将资产流程标记为已完成。

“完成任务”和“完成阶段”保持两个明确动作。完成最后一个任务不会在无提示的情况下跳走；点击“完成阶段”后，系统立即完成分配和流转。

忙闲程度直接使用现有数据计算：统计人员当前负责的进行中资产数量。不新增手工忙闲状态、不配置最大在办量，也不增加分配策略页面。

### 协作

- 协作任务不改变当前阶段、主负责人或主流程归属。
- 必做协作未完成时阻塞流转。
- 选做协作不阻塞流转；流转时自动取消仍未完成的选做协作，避免遗留跨阶段待办。
- 协作人员可以查看该客户和资产并处理自己的任务，但看不到“完成阶段”操作。

### 终止/流失

- 任意活跃流程都允许当前负责人执行“终止流程”。
- 必须填写原因。
- 终止后取消所有未完成待办，记录终止日志，并将进度标记为已终止；不再进入后续流程。

## 5. 默认业务配置

默认配置用于开箱演示，所有名称、部门、任务和顺序都可以修改：

### 签署流程

1. 接单建档：确认客户、建立最小资产、建群和首次沟通。
2. 资料收集：按资料模板持续补充资料，允许多次保存，不要求一次收齐。
3. 诊断核验：执行自动规则并展示核验结果。
4. 产品确认：确认客户适用的产品。
5. 合同签署：主负责人推进签署，律师、ALA 等通过必做协作任务参与。
6. 签约确认：登记签约结果，完成后进入运营流程。

线索不作为签署流程阶段。线索在转化前没有客户和资产，仍由独立线索池管理；转化动作负责创建客户、资产并分配首阶段负责人。

### 运营流程

运营阶段按实际租赁业务配置，例如待运营、招租、签租、交付、在租和退租。流程引擎不写死这些名称。

初始化只补齐缺失的示例配置，不删除客户、资产、资料或业务记录，也不保留旧复杂工作流的兼容分支。

## 6. 管理端界面

沿用当前简化布局：

- 顶部是流程标签，右上角新增、编辑、删除流程。
- 左侧是当前流程的阶段列表。
- 右侧是当前阶段的任务列表。
- 编辑流程时可配置“默认入口”和“完成后流程”。
- 编辑阶段时选择负责部门。
- 编辑任务时只展示当前任务类型需要的字段；资料任务显示资料模板，自动核验显示脚本规则，其他类型不显示无关配置。

不增加画布、连线、发布版本、条件配置或隐藏的高级模式。

## 7. 工作台界面

- “我的客户/资产”显示当前负责人是自己的业务。
- “协作任务”显示分配给自己的人员任务，以及分配给自己部门的部门任务。
- 详情显示当前流程、阶段、主负责人、必做任务完成进度和操作记录。
- 当前负责人在必做任务全部完成后看到一个主按钮“完成阶段”，点击后默认自动分配并流转。
- 主按钮旁保留“手动分配”，只有选择它时才打开人员选择弹窗；候选人限定为下一阶段负责部门的有效人员，并显示当前在办数量。
- 工作台不展示轮询、容量、忙闲状态或分配策略等配置项。
- 最后阶段自动按流程配置判断进入下一流程还是完成，不让业务人员临时选择任意流程。
- 当前负责人始终可以选择“终止流程”，但必须填写原因。

## 8. 数据与服务边界

在现有简化模型上就地调整，不新建另一套工作流：

- `Workflow` 增加默认入口标记和可空的后续流程 ID。
- `CustomerStage` 继续保存资产当前流程、阶段、部门和具体负责人，状态补充“已终止”及终止原因/时间。
- `WorkTodo` 继续承载主任务和协作任务，通过负责人方式区分，不再单建协作流程。
- 现有阶段变更日志继续记录进入、流转、完成和终止。

普通流程、阶段、任务配置继续使用 Model + page JSON。启动、任务完成、人员流转、终止涉及多表事务和权限，集中放在 workflow Service；work API 只做登录上下文、参数解析和响应。

## 9. 关键校验

- 活跃阶段必须配置有效负责部门。
- 只能存在一个启用的默认入口流程；默认入口必须至少有一个启用阶段。
- 活跃进度必须有具体负责人，且负责人属于当前阶段负责部门。
- 只能由当前负责人流转或终止。
- 自动分配只比较下一部门启用人员的进行中资产数量；数量相同时按最久未分配优先。
- 手动指定的下一负责人必须启用且属于下一阶段负责部门。
- 下一部门没有启用人员时保持原阶段不变，不创建部门共享待办。
- 存在未完成必做任务时拒绝流转，并返回具体未完成项。
- 流程链不能循环；停用或删除被引用流程时给出明确错误。
- 重复点击任务完成、流转或终止不能重复创建待办或重复推进。

## 10. 验收链路

1. MKT 在线索池转化线索，创建客户和最小资产；系统默认分配 NPL 接收人，也允许 MKT 手动指定。
2. 只有被分配的 NPL 是首阶段主负责人；他完成建档后点击“完成阶段”，系统默认分配下一阶段当前在办最少的人员，也允许手动指定。
3. 资料可以分多次填写，资料未齐时一直停留在当前阶段。
4. 资料满足后自动核验任务执行，但阶段不会自动前进。
5. 必做协作未完成时主负责人不能流转；协作完成不改变主负责人。
6. 主负责人完成产品、合同相关任务并逐阶段自动流转，每次都保留手动分配入口。
7. 签署流程最后阶段按配置进入运营流程，并自动分配运营首阶段具体负责人。
8. 任一阶段选择流失/终止后不进入后续流程，并能查看原因和日志。
9. 管理员可以新增、改名、排序、停用或删除未被使用的流程、阶段和任务，全程不接触条件规则或流程画布。
