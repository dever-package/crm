# 十一维 V4-RC1 完整落地设计

## 目标

依据 `doc/1.zip` 中的 Markdown 说明和 Excel 明细，将现有 P01-P12 十一维配置整体替换为 V4-RC1 定义的 P01-P11，并完整落地：

- 11 个固定输入维度和 114 个标准选项；
- 114 个选项的单项规则标签；
- 35 条跨维组合规则；
- 四层 T 节点记录；
- R0-R4 初判路由；
- 27 项初判输出；
- T 节点到 S 服务候选映射；
- 可回溯的规则执行快照。

本轮允许删除全部旧十一维数据，不做 V3 到 V4 的兼容映射。当前阶段只编写重建命令，不执行命令、不重启服务、不改变数据库。

## 设计原则

1. 复用现有 `DataTemplate`、`DataField`、`OptionSet`、`Form`、`RuleScript`、`DataRecord` 和 `OperationLog`。
2. 不新增十一维专用字段表、规则表、快照表或 CRUD API。
3. 固定业务字典和规则由独立 Go 重建命令通过现有 CRM Model 和 ORM 事务写入配置表，不编写 SQL migration。
4. 规则执行继续使用现有 JavaScript `RuleScript`，避免在 Go 代码中复制 35 条业务规则。
5. 系统只生成候选结论，不静默覆盖 NPL 输入、PM 正式 T 或人工产品结论。

## 方案选择

采用“事务重建命令 + 现有脚本规则 + 少量运行时适配”。不为 V4-RC1 新建规则配置模型。

Excel 是重建命令的数据来源，不是系统日常运行依赖。命令执行时从 `doc/1.zip` 读取并校验冻结数据，发布后的事实来源是数据库中的选项集、字段、表单和规则脚本。

## 数据清理边界

重建命令只清理旧十一维及其自动派生数据：

- 旧十一维 `DataRecord`；
- 旧十一维字段对应的 `StatFieldValue`；
- 旧十一维表单字段；
- P01-P12 分组、子字段、选项集及选项；
- 旧“十一维T节点自动判断”规则内容；
- 旧规则生成且仍为 `candidate` 的客户产品候选。

重建命令不得删除：

- 客户和线索；
- 客户资产；
- 已确认、处理中或已完成的客户产品；
- 与十一维无关的数据模板、表单、规则和操作记录。

旧十一维相关的操作记录和统计事件必须清理，但必须按旧表单任务、旧规则任务或关联数据记录精确过滤，不能按客户或资产整批删除。

## 新十一维输入

十一维本身就是 11 个独立的单选下拉字段，不再使用父分组和分组子字段：

| 编码 | 标准名称 | 选项数 |
| --- | --- | ---: |
| P01 | 案件情况 | 13 |
| P02 | 司法T节点 | 11 |
| P03 | 产权与签署主体状态 | 10 |
| P04 | 占有与腾房交付状态 | 13 |
| P05 | 产调与查封核验状态 | 8 |
| P06 | 债权及抵押结构 | 11 |
| P07 | 资产标的运营基础 | 9 |
| P08 | 特殊权利及敏感居住主体 | 9 |
| P09 | 房屋权利与用途性质 | 10 |
| P10 | 既有租赁与历史长租状态 | 10 |
| P11 | 隐性负担及欠费状态 | 10 |

11 个探针字段之后只保留一套公共补充字段：

1. 客户原话；
2. 事实摘要；
3. 证据状态；
4. 文件证据；
5. 缺失项；
6. 风险等级；
7. 复核备注。

新模板共 18 个平级字段：11 个探针下拉和 7 个公共补充字段。11 个探针下拉必填，公共补充字段可选。公共补充字段描述本次十一维采集的整体事实和证据，不分别归属于 P01-P11。

P12 客户配合度不再属于十一维，不创建对应字段。客户失联、补件、预约和沟通状态继续归属沟通、任务和日程数据。

## 选项标签和组合规则

114 个选项的输入状态、信息缺口、信息冲突、风险等级、路由下限、红线、专项预审、签约阻断、交付阻断和协同角色，作为 V4-RC1 规则脚本中的冻结配置写入。

35 条组合规则按以下顺序执行：

1. 红线覆盖；
2. T 节点路由；
3. 信息回退；
4. 跨维一致性；
5. 签约和交付门禁；
6. 专项会审；
7. 正向候选。

规则必须按 Excel 的优先级升序处理。最终路由覆盖顺序固定为 `R4 > R3 > R0 > R2 > R1`：资料缺口 R0 可以覆盖专项预审 R2，但不能覆盖暂缓 R3 或风险冻结 R4。低优先级正向规则不得降低已命中的路由。

## 四层 T 节点

诊断结果中分别保存：

- `npl_candidate_t`：直接读取 P02 的 T0-T10；
- `system_suggested_t`：根据 P04/P05 等事实给出的纠错建议；
- `routing_t`：路由使用的最低风险 T 节点；
- `formal_t`：PM/律师人工确认的正式 T 节点。

自动规则只能写前三项。`formal_t` 必须由人工确认表单写入，不能由规则脚本回填。

## 27 项输出

在“诊断结果”数据模板中配置文档定义的 OUT-01 至 OUT-27。字段类型复用现有 `text`、`number`、`select`、`multi_select` 和 `textarea`。

规则脚本返回：

- 完成度和有效可判定度；
- 候选 T、建议 T、路由 T 和阶段带；
- 案件、产权、占有、查封、债权、资产、租赁等摘要；
- 风险标签和命中规则；
- R0-R4 路由；
- S 服务候选及资产运营产品门禁；
- 补件、协同、下一动作和禁止动作；
- 规则版本与发布版本。

OUT-01 判断快照编号由服务端在操作记录创建后生成，不能由 JavaScript 使用时间戳伪造唯一编号。

## R0-R4 与流程门禁

R0-R4 是初判结果，不直接等于客户主状态，也不直接批准正式产品或合同。

- R0：补探针/补证据；
- R1：常规 PM 核验；
- R2：专项人工预审；
- R3：暂缓/解卡后重评；
- R4：风险冻结/不可接入候选。

现有“诊断核验”阶段继续承担人工门禁。阶段任务调整为：

1. 自动运行 V4-RC1 初判；
2. 确认诊断结果：同一审批任务填写正式 T、审核结果和审核意见。

审批通过时，正式 T、审批操作记录和待办状态在同一事务中保存；审批驳回时不覆盖正式 T，并保留当前待办或按后台配置回流。只有合并审批通过后，流程才进入产品确认。规则本身不直接推动签约。

## T 到 S 服务候选

规则按 Excel `12_T节点服务映射_规则` 输出候选产品编码。候选结果继续使用现有 `SyncCandidateCustomerProducts` 保存，人工产品确认仍是最终决定。

重建命令补充当前产品库缺失的 `S02` 客户案件基础画像服务。T7-T8 冻结新增资产运营产品，T9-T10 只保留 S21 和存量收尾；结案归档工作包不创建 S 产品编号。

已确认、处理中和已完成的产品不因新规则重算而删除或降级。

## 判断快照

复用 `OperationLog`，不新增快照表。每次规则执行保存：

- 唯一快照编号；
- 客户、资产、流程实例和任务；
- 完整 P01-P11 输入；
- 规则输出和命中规则；
- 规则版本、发布版本和执行耗时；
- 执行时间。

后续修改当前资料或升级规则不得覆盖历史操作记录。判断快照编号使用已落库的操作记录 ID 生成，操作快照中的输入只保存本次 P01-P11 字段值，不复制客户或资产的其它资料。

## 页面和统计

工作台继续使用现有动态表单，不新建十一维专用录入页面。新模板不包含 `group` 字段，因此不得再生成左侧 P01-P11 分组导航。表单按平级字段顺序展示：先显示 11 个探针下拉，再显示 7 个公共补充字段。

统计只计算 11 个探针下拉，不把公共补充字段计入十一维完成度：

- P01-P12 改为 P01-P11；
- 完整度分母使用 11；
- 11 个探针全部选择后才允许确认完成；
- 诊断结果通过现有资料详情展示。

## 重建与发布

新增 `cmd/rebuild-eleven-dimension` Go 命令。命令默认仅执行文档校验和数据库预检；只有显式传入 `--apply` 时才在单个 ORM 事务内清理并重建。缺少旧模板、目标分类、诊断模板、签约流程或关键任务时，命令应主动报错并回滚，不能留下半套配置。

当前阶段只提交命令文件，不执行命令。后续执行前应先备份数据库，由用户在允许的窗口手动验证。

## 验证边界

项目明确禁止运行 build 和任何 test，因此实现期间不执行 `npm run build`、`go test`、`dever build` 或迁移演练。

允许的静态检查仅包括：

- Go 命令人工审阅；
- `git diff --check`；
- Dever 静态审计脚本；
- 对照 Excel 统计 11 个维度、114 个选项、35 条规则和 27 个输出定义。
