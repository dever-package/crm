# CRM 后台线索新增与产品展示设计

## 目标

- 后台“线索信息”页支持新增线索。
- 产品列表正确展示产品分类。
- 将产品的“服务流程”改为更准确的“签约后流程”，不改变数据结构和运行逻辑。

## 边界

线索新增不是普通单表 CRUD。它必须继续执行现有的线索编号生成、联系方式校验、来源渠道校验和重复检测。后台不另写一套规则，而是与工作台共享登记能力。

产品的签约后流程为可选配置：

- 不选择：产品在入口签约流程完成后结束。
- 选择：签约完成后自动进入指定流程，用于租赁等后续运营。

## 实现设计

1. 抽取一个与站点身份无关的线索登记输入和结果结构，将现有 `WorkService.RegisterLead` 中的事务逻辑收敛到共享方法。
2. 工作台保持现有 API 和员工上下文，调用共享方法。
3. 后台新增薄 API 负责管理端身份适配，再调用同一共享方法。管理员能关联到 CRM 员工时记录创建人和初始负责人；无员工关联时保持未分配。
4. 后台线索列表增加“新增线索”弹窗，提交核心字段和线索数据模板字段，成功后刷新列表。
5. 产品列表将分类展示路径由错误的 `product_category.name` 改为 Model Relation 实际输出的 `category.name`。
6. 产品列表、编辑表单和校验提示统一使用“签约后流程”文案，底层仍使用 `service_workflow_id`，避免无意义的数据迁移。

## 异常处理

- 姓名为空，或手机和微信同时为空时拒绝保存。
- 来源、渠道不存在或已停用时拒绝保存。
- 检测到重复时仍创建“重复”状态线索，与工作台行为一致。
- 签约后流程仍只允许选择已启用的非入口流程，并保留首阶段自动分配校验。

## 验证

只执行 Dever 静态审计、JSON 解析和变更检查。按项目要求，不执行 build 或 test，由用户手动验证管理端新增、重复线索标记、产品分类展示和签约后流程保存。
